# 🎵 Music Generation AI

An end-to-end pipeline that **learns musical patterns** from your MIDI dataset
and composes entirely new pieces — using LSTM networks and the music21 library.

---

## ✨ Features

| Feature | Details |
| --- | --- |
| **MIDI Parsing** | Handles single notes, chords, and multi-track files via music21 |
| **Pitch-aware Encoding** | Notes and chords tokenised with full octave information |
| **LSTM + Embedding** | Two stacked LSTM layers with BatchNorm and Dropout |
| **Temperature Sampling** | Tune creativity vs. predictability at generation time |
| **Smart Callbacks** | Auto-saves the best checkpoint; reduces LR on plateau; early stops |
| **Single Config** | Every hyperparameter lives in `config.py` |

## 🎼 Dataset Preparation

### Where to put your files

Drop all your MIDI files into **`data/midi/`**.
Subdirectories are fully supported — the loader recurses automatically.

```
data/
└── midi/
    ├── classical/
    │   ├── bach_invention_1.mid
    │   └── chopin_nocturne.mid
    └── jazz/
        └── autumn_leaves.mid
```

### Where to get MIDI datasets

| Source | Genre | Notes |
|---|---|---|
| [Magenta MAESTRO](https://magenta.tensorflow.org/datasets/maestro) | Classical Piano | 200 h, research-grade |
| [Classical MIDI Archives](https://www.classicalarchives.com) | Classical | Large catalogue |
| [BitMidi](https://bitmidi.com) | Mixed | Free download |
| [FreeMIDI](https://freemidi.org) | Mixed | Free download |
| Your own recordings / DAW | Any | Export as MIDI from GarageBand, FL Studio, etc. |

> **Quality tip** — More data = better results.
> Aim for at least **50–100 MIDI files** from the same genre for coherent output.
> Mixing very different genres (e.g. classical + death metal) confuses the model.

## 🛠 Usage — Step by Step

All pipeline stages are available through the unified `main.py` CLI
or by running each script directly.

---

### Step 1 — Preprocess

Parse your MIDI files, extract notes and chords, and build the vocabulary.

```bash
python main.py preprocess
```

**What it creates inside `data/processed/`:**

| File | Contents |
| --- | --- |
| `elements.pkl` | List of `(token, duration)` tuples from all MIDI files |
| `note_to_int.pkl` | Mapping: token string → integer index |
| `int_to_note.pkl` | Mapping: integer index → token string |

> You only need to run this **once per dataset**.
> Add `--force-preprocess` to the `train` command to re-parse after adding new files.

---

### Step 2 — Train

```bash
python main.py train
```

If cached data doesn't exist yet, preprocessing runs automatically.
To force re-parsing:

```bash
python main.py train --force-preprocess
```

**What happens during training:**

- Sequences of length `SEQUENCE_LENGTH` slide over all your note data
- The LSTM learns to predict the next note / chord given the previous N notes
- **`models/best_model.keras`** — saved whenever loss improves
- **`models/final_model.keras`** — saved at the end of training
- **`models/training_history.png`** — loss and accuracy curves

**Callbacks that run automatically:**

| Callback | Behaviour |
| --- | --- |
| `ModelCheckpoint` | Saves the best model whenever training loss improves |
| `EarlyStopping` | Stops training if loss doesn't improve for 10 epochs |
| `ReduceLROnPlateau` | Halves the learning rate if loss stalls for 5 epochs |

---

### Step 3 — Generate

```bash
python main.py generate
```

Advanced options:

```bash
python main.py generate \
  --model      models/best_model.keras \
  --output     output/my_song.mid \
  --length     500 \
  --temperature 1.0
```

| Flag | Default | Description |
|---|---|---|
| `--model` | `models/best_model.keras` | Which checkpoint to load |
| `--output` | `output/generated.mid` | Where to save the MIDI |
| `--length` | `500` (from config) | Number of notes / chords to generate |
| `--temperature` | `1.0` (from config) | Creativity level — see table below |

## 🔧 Configuration Reference

Edit `config.py` to tune any parameter without touching the code:

```python
# config.py

MIDI_DIR   = "data/midi"      # Source MIDI directory
DATA_DIR   = "data/processed" # Preprocessed cache location
MODELS_DIR = "models"         # Model checkpoint directory
OUTPUT_DIR = "output"         # Generated MIDI directory

SEQUENCE_LENGTH = 100         # Context window (past notes the model sees)
EMBEDDING_DIM   = 64          # Size of the token embedding vector
LSTM_UNITS      = 512         # Units per LSTM layer
DROPOUT_RATE    = 0.3         # Regularisation dropout

BATCH_SIZE    = 64            # Training mini-batch size
EPOCHS        = 100           # Maximum training epochs
LEARNING_RATE = 0.001         # Adam initial learning rate

GENERATE_LENGTH = 500         # Notes to generate
TEMPERATURE     = 1.0         # Sampling temperature
BPM             = 120         # Output MIDI tempo (beats per minute)
```

## 🧠 Model Architecture

```
Input  (batch, 100)              Integer token IDs (one per timestep)
   │
   ▼
Embedding  (n_vocab → 64)        Dense learned vector per token
   │
   ▼
LSTM₁  (512 units, seq=True)   ─┐  Captures short-term melodic patterns
Dropout(0.3) + BatchNorm         │
   │                             │
   ▼                             │
LSTM₂  (512 units, seq=False)  ─┘  Compresses to one context vector
Dropout(0.3) + BatchNorm
   │
   ▼
Dense(256, relu)
Dropout(0.3)
   │
   ▼
Dense(n_vocab, softmax)          Probability over every possible next token
```

**Why Embedding instead of raw normalised integers?**
The Embedding layer lets the model learn a *dense geometric relationship* between notes — so "C4" and "C5" can be close in embedding space, capturing musical intuition the model builds from data.

**Why sparse_categorical_crossentropy?**
The target `y` is a single integer (the next token index), so there's no need to one-hot encode it — `sparse_categorical_crossentropy` handles it directly and saves memory.

## 🎛 Temperature Guide

| Temperature | Character | Use case |
|---|---|---|
| `0.5` | Very repetitive, stays close to training patterns | Predictable / background music |
| `0.7` | Conservative with small variations | Classical-style, structured |
| `1.0` | Balanced — default | General purpose |
| `1.2` | More adventurous, occasional surprises | Jazz-style improvisation |
| `1.5` | Very experimental, may sound chaotic | Creative / abstract exploration |

**Tip:** Generate 3–5 files at different temperatures and keep the best ones.

## 💡 Tips for Better Results

### Data quality
- **Volume** — 100+ MIDI files in the same genre produce noticeably better output
- **Consistency** — Keep genres uniform; mixed datasets produce muddled style
- **Cleanup** — Remove MIDI files that produce `[WARN]` parse errors

### Training
- Monitor **`models/training_history.png`** after training
- If the loss curve is still dropping at epoch 100, increase `EPOCHS` in config
- If training loss oscillates, lower `LEARNING_RATE` to `0.0005`
- On CPU: reduce `LSTM_UNITS` to `256` and `BATCH_SIZE` to `32` for speed

### Generation
- Longer sequences (`--length 800+`) often sound like complete musical phrases
- Run generation multiple times — each run picks a different random seed
- Use a DAW (GarageBand, FL Studio, LMMS) to add reverb and mix the output

## 🐛 Troubleshooting

| Problem | Fix |
|---|---|
| `No MIDI files found in 'data/midi'` | Verify your `.mid` / `.midi` files are inside `data/midi/` |
| `Vocabulary size: 0` | All MIDI files failed parsing — check for corruption |
| `OOM (Out of Memory)` during training | Reduce `BATCH_SIZE` to 32 or 16; reduce `LSTM_UNITS` to 256 |
| Very slow training | Enable GPU — install `tensorflow[and-cuda]` |
| `FileNotFoundError` on generate | Run `preprocess` and `train` steps first |
| Generated music sounds random / noisy | Train more epochs; use more data; try lower temperature (0.7) |
| music21 `[WARN]` messages during parse | A few corrupted / unsupported files is fine — remove them if many |
| `module not found` error | Activate your virtual environment and re-run `pip install -r requirements.txt` |
